# Frog Chess Server

Серверная часть многопользовательской игры Frog Chess, реализованная на платформе .NET 10 с использованием SignalR.

## Описание проекта

Frog Chess — пошаговая игра для двух игроков на доске 8×8, в которой лягушки могут прыгать через других лягушек.
Сервер является единственным источником истины и полностью отвечает за:

* создание и хранение игровых сессий;
* генерацию начальной расстановки;
* проверку всех правил игры;
* синхронизацию состояния между игроками;
* определение победителя;
* обработку отключений и переподключений.

Все игровые данные хранятся в памяти приложения. После перезапуска сервера все активные игры удаляются.

---

# Архитектурное решение

Для проекта был выбран **вариант А — чистый SignalR**.

Все взаимодействие клиента с сервером происходит через SignalR Hub:

* создание игры;
* подключение к игре;
* удаление лягушки на первом ходе;
* выполнение ходов;
* пропуск хода;
* получение обновлений состояния игры.

Причины выбора:

* игра требует мгновенного обмена сообщениями между двумя игроками;
* отсутствует необходимость в дополнительном HTTP API;
* SignalR автоматически поддерживает постоянное соединение и двустороннюю связь;
* архитектура получается проще и содержит меньше инфраструктурного кода.

От гибридного подхода было решено отказаться, поскольку REST не давал заметных преимуществ для небольшой пошаговой игры и потребовал бы дополнительного механизма идентификации игроков.

От подхода с polling было решено отказаться из-за лишнего сетевого трафика, задержек обновления состояния и менее удобного пользовательского опыта.

Для данной задачи использование чистого SignalR является компромиссом между простотой реализации и функциональностью.

Содержит основные игровые сущности:

* Game
* Board
* Player
* Frog
* Position

## Services

Содержит бизнес-логику приложения:

* GameService — управление игровым процессом;

* BoardGenerator — генерация стартовой расстановки;

Вспомогательные сервисы:

## Hubs

* GameHub — точка взаимодействия клиентов с сервером через SignalR.

## DTOs

Объекты передачи данных между сервером и клиентом.

# Игровой процесс

## Создание игры

Игрок вызывает метод:

CreateGame(playerName)

Сервер:

* создаёт игровую сессию;
* создаёт первого игрока;
* назначает ему зелёный цвет;
* возвращает идентификатор игры и токен игрока.

---

## Подключение к игре

Игрок вызывает:

JoinGame(gameId, playerName)

Сервер:

* добавляет второго игрока;
* назначает ему оранжевый цвет;
* случайным образом генерирует стартовую расстановку;
* переводит игру в состояние InProgress;
* отправляет состояние игры обоим игрокам.

---

## Первый ход

Перед первым прыжком каждый игрок обязан удалить одну любую лягушку.

Удаление возможно:

* только один раз;
* только до первого прыжка;
* только в свой ход.

После удаления ход автоматически переходит к сопернику.

---

## Выполнение хода

Игрок отправляет последовательность позиций:

MakeMove(gameId, token, path)

Сервер:

1. Проверяет очередь хода.
2. Проверяет выполненность первого удаления.
3. Проверяет корректность каждого прыжка.
4. Удаляет перепрыгнутых лягушек.
5. Перемещает лягушку.
6. Проверяет механику болота.
7. Передаёт ход сопернику.
8. Выполняет автоматический пропуск при отсутствии допустимых ходов.

---

## Пропуск хода

Если у игрока отсутствуют допустимые ходы, сервер автоматически выполняет пропуск.

Игрок также может вызвать PassTurn(gameId, token), если такая функция включена.

Игра завершается после двух последовательных пропусков.

---

## Определение победителя

Победителем считается игрок, который выполнил последний прыжок перед окончанием партии.

Если оба игрока подряд не имеют допустимых ходов, игра завершается автоматически.

Если (вдруг) игра совершилась ещё до прыжков, то засчитывается ничья.

---

## Механика болота

Крайние клетки доски являются болотом.

Если лягушка завершает ход в болоте, она немедленно удаляется с доски.

Если путь лягушки проходит через болото, но заканчивается на обычной клетке, она остаётся на доске.

---

# Переподключение

При отключении игрока:

* соединение помечается как разорванное;
* второй игрок получает уведомление.

Если игрок подключается повторно, используя то же имя:

* восстанавливается соединение;
* игрок возвращается в текущую партию;
* состояние игры повторно отправляется клиенту.

Если игрок не переподключился в течение заданного времени, победа присуждается сопернику.

---

# Защита от конкурентного доступа

Несколько клиентов могут одновременно отправлять команды в одну игру.

Для предотвращения повреждения состояния используется:

SemaphoreSlim

Каждая игра имеет собственный семафор, получаемый через сервис GameLockProvider.

Это гарантирует, что в каждый момент времени только одна операция изменяет состояние конкретной игры.

# Логирование

Сервер ведёт журнал следующих событий:

* создание игры;

Логи позволяют полностью восстановить жизненный цикл партии без изучения исходного кода.

# Документация SignalR API

## CreateGame

Вызов:

CreateGame(string playerName)

Ответ:

GameCreated(gameId, playerToken, color)

---

## JoinGame

Вызов:

JoinGame(string gameId, string playerName)

Ответ:

JoinedGame(gameId, playerToken, color)

---

## RemoveInitialFrog

Вызов:

RemoveInitialFrog(gameId, token, row, col)

---

## MakeMove

Вызов:

MakeMove(gameId, token, path)

---

## PassTurn

Вызов:

PassTurn(gameId, token)

---

# События сервера

## GameStarted

Отправляет начальное состояние игры.

## BoardUpdated

Отправляет новое состояние игры после любого изменения.

## GameFinished

Отправляет имя победителя.

## PlayerDisconnected

Уведомление об отключении игрока.

## PlayerReconnected

Уведомление о переподключении игрока.

# Тестирование

Для проекта используются модульные тесты на базе xUnit.

Покрыты следующие области:

* генерация стартовой расстановки;
* первое удаление лягушки;
* повторное удаление;
* удаление после первого прыжка;
* допустимые и недопустимые прыжки;
* цепочки прыжков;
* прыжки через своих и чужих лягушек;
* прыжки из болота;
* автоматический пропуск хода;
* определение победителя;
* завершение игры;
* переподключение игроков;
* конкурентный доступ.

# Запуск проекта

1. Запустить сервер:

dotnet run --project Server

2. Открыть клиентское приложение Frog Chess.

3. Создать игру и передать идентификатор второму игроку.

4. Подключиться ко второй стороне и начать игру.
